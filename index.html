What's Test Driven Development ?

  1. Why should we write tests first?

    If we write it at the end they won't be complete!

  2. It's just human nature...

    The most important things always come first... the ones at the end are, often, optional or undesirable.

First Law of TDD

You’re not allowed to write any production code until you’ve managed to write a failing Unit Test.

Second Law of TDD

You can’t write more than an Unit Test than what’s sufficient to fail

Third Law of TDD

You’re not allowed to write more production code than the sufficient to pass a failing test

Laws of TDD

  • Write the next test.

    While several tests may be outlined, only one is written at a time.

    In pessimistic languages, add enough stubs so the test compiles.

    It should fail when run.

    Refactor, if necessary, to prepare for the implementation.

  • Change the logic of the system so the test passes.

  • Refactor to eliminate duplication and other excess complexity.

Kent Beck

Conclusions

Low Debugging Time

  1. Usually means low productivity
  2. We don’t wan’t people to spend time debugging
  3. We want people to spend time writing production ready code

Conclusions

Design Documents

  1. TDD gives us really good and useful (development perspective) low level specification.
  2. Always updated.

Conclusions

Decoupling

  1. Writing tests first infuences the way we write code.
  2. Writing tests first leads to a better design... highly decoupled code.

Conclusions

Courage to Change

  1. Tests eliminate the fear of change
  2. Tests stop the code from rotten
